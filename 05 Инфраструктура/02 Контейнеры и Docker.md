## Классическая проблема

"У меня работает, на сервере не работает."

Причина: на ноутбуке Java 21, на сервере Java 11. Или разные версии библиотек. Или другая ОС. Окружение разработки и production различаются.

**Docker** решает это: упакуй приложение вместе с окружением.

## Контейнер vs Виртуальная машина

```
Виртуальная машина:          Контейнер:
┌─────────────────┐          ┌─────────────────┐
│   Приложение    │          │   Приложение    │
├─────────────────┤          ├─────────────────┤
│   Гостевая ОС   │          │  Зависимости    │
│  (2GB+, минуты) │          │ (МБ, секунды)   │
├─────────────────┤          ├─────────────────┤
│  Гипервизор     │          │    Docker       │
├─────────────────┤          ├─────────────────┤
│  Хостовая ОС    │          │  Хостовая ОС    │
└─────────────────┘          └─────────────────┘
```

Контейнер не содержит целую ОС — он использует ядро хостовой ОС. Поэтому лёгкий и быстрый.

## Dockerfile — рецепт контейнера

```dockerfile
FROM openjdk:21-jre        # базовый образ
WORKDIR /app               # рабочая директория
COPY app.jar .             # копируем приложение
EXPOSE 8080                # открываем порт
CMD ["java", "-jar", "app.jar"]  # команда запуска
```

```bash
docker build -t myapp .    # собрать образ
docker run -p 8080:8080 myapp  # запустить контейнер
```

## Kubernetes — управление контейнерами в масштабе

Когда контейнеров сотни — нужен оркестратор:

- **Kubernetes (K8s)** — запускает нужное количество копий каждого сервиса, перезапускает упавшие, обновляет без downtime

```
Kubernetes кластер
├── Node 1: [UserService] [AuthService]
├── Node 2: [UserService] [OrderService]
└── Node 3: [OrderService] [PaymentService]
```
