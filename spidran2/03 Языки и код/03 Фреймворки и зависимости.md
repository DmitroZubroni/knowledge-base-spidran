## Никто не пишет всё с нуля

Когда тебе нужно отправить email из приложения — ты не изучаешь стандарт SMTP и не пишешь код обмена пакетами. Ты берёшь готовую **библиотеку** которая это уже умеет.

Разработка на 80% — это сборка из готовых компонентов. Задача — соединить их правильно.

## Библиотека vs Фреймворк

| | Библиотека | Фреймворк |
|-|-----------|-----------|
| Кто управляет | Ты вызываешь библиотеку | Фреймворк вызывает твой код |
| Пример | Jackson (парсинг JSON) | Spring Boot (весь бэкенд) |
| Гибкость | Высокая | Ниже — нужно следовать структуре |
| Принцип | "Возьми что нужно" | "Вставь своё в готовый каркас" |

**Аналогия:** библиотека — это инструмент (дрель). Фреймворк — это строительная бригада у которой есть свой процесс, и ты работаешь по их правилам.

## Менеджеры зависимостей

Проект использует десятки библиотек. Хранить их в репозитории — плохая идея. Вместо этого — файл с описанием **зависимостей**:

| Экосистема | Файл | Команда загрузки |
|-----------|------|-----------------|
| Java (Maven) | `pom.xml` | `mvn install` |
| Java (Gradle) | `build.gradle` | `gradle build` |
| Node.js | `package.json` | `npm install` |
| Python | `requirements.txt` | `pip install -r requirements.txt` |

В файле указываешь имя библиотеки и версию. Менеджер скачивает всё необходимое.

## Версионирование библиотек — semver

Версии вида `2.4.1` означают:

```
2   .   4   .   1
↑       ↑       ↑
major  minor  patch

patch: исправление багов
minor: новые функции (обратная совместимость)
major: ломает совместимость
```

Обновление patch и minor — обычно безопасно. Обновление major — читай список изменений.
