## Монолит — один большой сервис

**Монолит** — всё приложение в одном деплоименте. Авторизация, каталог товаров, оплата, уведомления — всё в одном запущенном процессе.

```
[Монолит]
  ├── UserModule
  ├── ProductModule
  ├── OrderModule
  └── NotificationModule
        ↕
   [База данных]
```

**Плюсы:** просто разрабатывать, просто деплоить, просто отлаживать.  
**Минусы:** при росте становится сложным. Одна упавшая фича валит всё. Масштабировать можно только целиком.

---

## Микросервисы — много маленьких сервисов

**Микросервисы** — каждая функциональность это отдельный сервис со своей БД и деплойментом.

```
[User Service]    [Product Service]    [Order Service]
      ↕                  ↕                   ↕
  [Users DB]       [Products DB]        [Orders DB]
       ↕                  ↕                   ↕
           [API Gateway] ← клиенты идут сюда
```

**Плюсы:** масштабировать только нужные части, технологическая свобода, независимые деплойменты.  
**Минусы:** сложность инфраструктуры, сетевые вызовы, распределённые транзакции.

---

## Когда что подходит

| Ситуация | Выбор |
|----------|-------|
| Стартап, MVP, маленькая команда | Монолит — быстрее запуститься |
| Много команд, разные домены | Микросервисы |
| Одна часть нагружена сильно | Выделить её в микросервис |

**Правило:** начинай с монолита. Переходи на микросервисы когда монолит реально болит.

Netflix, Uber, Amazon используют микросервисы. Но они начинали с монолита.

---

## 🎯 Главное

> Монолит — проще для старта. Микросервисы — гибче при масштабировании. Большинство команд переходят на микросервисы по мере роста, а не с первого дня.
