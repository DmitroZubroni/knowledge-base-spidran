## Зачем вообще версионирование

Представь: ты написал код, всё работало. Добавил новую фичу — всё сломалось. А ты уже 20 раз сохранил файл и назад не вернуться.

**Git** решает эту проблему. Он хранит историю всех изменений. Ты можешь вернуться к любому моменту.

---

## Основные концепции

**Репозиторий (repo)** — папка с кодом + вся история изменений.

**Коммит (commit)** — снимок состояния кода в момент времени с описанием:
```
commit a3f5b2c
Author: Дмитрий
Date:   Mon Jun 23

    feat: добавил авторизацию через JWT
```

**Ветка (branch)** — параллельная линия разработки. Пока ты делаешь новую фичу в отдельной ветке, `main` остаётся стабильным.

---

## Типичный рабочий процесс

```
main ─────●──────────────────●─── (стабильный релиз)
           \                /
feature ────●──●──●──●──●──  (разработка фичи)
```

1. Создал ветку: `git checkout -b feature/login`
2. Написал код, сделал коммиты
3. Открыл **Pull Request** — просишь команду проверить код
4. После проверки — влил (merge) в `main`

---

## GitHub — облако для кода

**GitHub** (и GitLab, Bitbucket) — это хостинг для git-репозиториев. Там:
- Хранится код удалённо
- Команда видит чужие изменения
- Идут code review (проверки кода)
- Запускается автоматизация (CI/CD)

---

## Базовые команды

```bash
git clone <url>         # скачать репозиторий
git pull                # получить последние изменения
git add .               # добавить все изменения
git commit -m "message" # сохранить снимок
git push                # отправить на GitHub
git checkout -b <name>  # создать ветку
```

---

## 🎯 Главное

> Git хранит историю кода. Ветки — для параллельной работы. GitHub — облако для совместной работы. Без Git в команде — хаос.
