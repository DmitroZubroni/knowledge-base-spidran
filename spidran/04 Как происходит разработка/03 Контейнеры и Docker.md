## Классическая проблема

"У меня работает, а на сервере не работает."

Причина: на твоём ноутбуке Java 17, на сервере Java 11. Или разные версии библиотек. Или другая ОС.

**Docker** решает это: упакуй приложение **вместе со всем окружением** в контейнер.

---

## Контейнер — изолированная среда

**Контейнер** — это изолированный процесс со своей файловой системой, зависимостями и настройками.

**Аналогия:** контейнер как рыболовный ящик. Всё что нужно для рыбалки — уже внутри. Берёшь ящик → идёшь рыбачить. Берёшь Docker-контейнер → запускаешь приложение.

```dockerfile
FROM openjdk:17-jre
COPY app.jar /app.jar
CMD ["java", "-jar", "/app.jar"]
```

Этот `Dockerfile` — рецепт контейнера. `docker build` строит образ. `docker run` запускает.

## Docker vs виртуальная машина

| | Виртуальная машина | Docker-контейнер |
|-|--------------------|------------------|
| Размер | Гигабайты | Мегабайты |
| Запуск | Минуты | Секунды |
| Изоляция | Полная ОС | Процесс в ОС хоста |
| Накладные расходы | Высокие | Минимальные |

---

## Kubernetes — оркестрация контейнеров

Когда контейнеров много (сотни микросервисов) — нужен **Kubernetes (K8s)**:
- Запускает нужное количество копий каждого сервиса
- Перезапускает упавшие контейнеры
- Балансирует нагрузку между копиями
- Обновляет без downtime

Kubernetes = диспетчер для контейнеров.

---

## 🎯 Главное

> Docker упаковывает приложение со всем окружением → "работает везде". Kubernetes управляет многими контейнерами в production. Без Docker деплой хрупкий и зависит от настройки сервера.
