## Почему не хранить всё в файлах?

Можно. Но представь что у тебя миллион пользователей. Искать нужного в текстовом файле — это перебрать всех. БД умеют искать за миллисекунды благодаря **индексам** — специальным структурам для быстрого поиска.

---

## Реляционные БД — таблицы и связи

**SQL-базы** (PostgreSQL, MySQL, SQLite) хранят данные в таблицах — как Excel.

```
Таблица users:
| id | name     | email              |
|----|----------|--------------------|
| 1  | Дмитрий  | d@example.com      |
| 2  | Мария    | m@example.com      |

Таблица orders:
| id | user_id | product   | price |
|----|---------|-----------|-------|
| 1  | 1       | Курс Java | 5000  |
| 2  | 1       | Книга     | 800   |
```

`user_id` — это **внешний ключ**, связывающий заказ с пользователем. Отсюда "реляционная" (от relation — связь).

**SQL** — язык запросов к таким БД:
```sql
SELECT * FROM orders WHERE user_id = 1;
-- Найти все заказы пользователя с id=1
```

## NoSQL — гибкость за счёт структуры

**NoSQL-базы** (MongoDB, Redis, Cassandra) не требуют фиксированной структуры таблиц.

**MongoDB** хранит документы (JSON-подобные объекты):
```json
{
  "id": 1,
  "name": "Дмитрий",
  "orders": [
    { "product": "Курс Java", "price": 5000 }
  ]
}
```

**Redis** — хранит данные в памяти (очень быстро). Используется для кеша и сессий.

---

## Когда что использовать

| Ситуация | Выбор |
|----------|-------|
| Структурированные данные, связи | PostgreSQL / MySQL |
| Гибкая схема, большие объёмы документов | MongoDB |
| Кеширование, очереди, счётчики | Redis |
| Аналитика, хранилище событий | ClickHouse, Cassandra |

В большинстве бэкенд-проектов — PostgreSQL.

---

## 🎯 Главное

> БД нужна для надёжного хранения данных с быстрым поиском. SQL-базы — таблицы со связями. NoSQL — гибкие документы или ключ-значение. PostgreSQL — выбор по умолчанию.
