## Аутентификация и авторизация

Две разные вещи:

- **Аутентификация** — *кто ты?* (логин + пароль, Google OAuth, Face ID)
- **Авторизация** — *что тебе можно?* (admin видит всё, user — только своё)

Сначала аутентификация, потом авторизация.

## Как работают токены (JWT)

Современные приложения не хранят сессию на сервере. Используют **JWT (JSON Web Token)**:

```
1. Пользователь вводит логин/пароль
2. Сервер проверяет → выдаёт JWT-токен
3. Токен хранится в браузере (cookie / localStorage)
4. Каждый запрос содержит токен:
   Authorization: Bearer eyJhbGc...
5. Сервер проверяет подпись → знает кто это без БД
```

JWT — строка из трёх частей: `header.payload.signature`. Payload содержит id пользователя и роли.

## Хеширование паролей

Пароли **никогда** не хранятся в открытом виде:

```
"password123" → bcrypt → "$2a$10$N9qo8uLOic..."
```

Хеш нельзя расшифровать. При проверке: хешируется введённый пароль, сравнивается с сохранённым хешем.

## HTTPS и TLS

Без HTTPS все данные между клиентом и сервером идут в открытом виде — провайдер, хакер в сети или любой промежуточный узел может их прочитать.

**TLS** шифрует соединение. HTTPS = HTTP + TLS.

Сертификаты выдаёт **Certificate Authority (CA)** — доверенная организация. Let's Encrypt выдаёт бесплатно.

## CORS — защита браузера

Браузер запрещает сайту `evil.com` делать запросы к `bank.com` от твоего имени.

Если бэкенд хочет разрешить запросы с конкретного домена:
```
Access-Control-Allow-Origin: https://myapp.com
```

Ошибка CORS в консоли — бэкенд не добавил это разрешение.
