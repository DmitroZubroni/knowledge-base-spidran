## Зачем писать тесты если "и так работает"

"И так работает" — сейчас. Через месяц ты (или коллега) изменишь одну строку и сломаешь три другие функции, не зная об этом.

**Тесты** — это код, который проверяет другой код. Они ловят поломки автоматически.

---

## Пирамида тестирования

```
        /\
       /  \     E2E тесты (мало, дорогие)
      /────\    Интеграционные (средне)
     /      \   Юнит-тесты (много, быстрые)
    /────────\
```

**Юнит-тест** — тестирует одну функцию изолированно:
```java
@Test
void calculateDiscount_whenVIP_returns20percent() {
    User vip = new User(Role.VIP);
    assertEquals(20, discountService.calculate(vip));
}
```

**Интеграционный** — тестирует несколько компонентов вместе (например, сервис + БД).

**E2E (End-to-End)** — тестирует всё приложение как пользователь (Selenium, Playwright).

---

## TDD — сначала тест, потом код

**TDD (Test-Driven Development):**
1. Напиши тест который падает (функции ещё нет)
2. Напиши минимальный код чтобы тест прошёл
3. Отрефактори

Звучит странно, но это дисциплинирует думать о поведении кода до его написания.

---

## Покрытие кода

**Code Coverage** — процент кода, покрытого тестами. 80% — хорошая цель. 100% — часто избыточно.

В CI пайплайне тесты запускаются автоматически при каждом коммите.

---

## 🎯 Главное

> Тесты ловят поломки автоматически. Юнит-тесты — быстрые и изолированные. CI запускает их при каждом коммите. Без тестов рефакторинг = русская рулетка.
