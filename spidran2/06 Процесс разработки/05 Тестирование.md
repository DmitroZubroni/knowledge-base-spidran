## Зачем тесты если "и так работает"

Сейчас — работает. Через месяц ты или коллега изменит строку кода и незаметно сломает три другие функции. Без тестов узнаешь об этом от пользователей в production.

**Тесты** — код который автоматически проверяет другой код.

---

## Пирамида тестирования

```
        /\
       /  \
      / E2E \        Мало, медленные, дорогие
     /────────\
    /Integration\    Средне
   /────────────\
  /  Unit Tests  \   Много, быстрые, дешёвые
 /────────────────\
```

**Юнит-тест** — тестирует одну функцию изолированно:

```java
@Test
void calculateDiscount_whenVIP_returns20() {
    // Arrange
    User vip = new User(Role.VIP);

    // Act
    int discount = discountService.calculate(vip);

    // Assert
    assertEquals(20, discount);
}
```

**Интеграционный тест** — проверяет взаимодействие нескольких компонентов (например, сервис + БД).

**E2E тест** — имитирует реального пользователя в браузере (Playwright, Selenium).

---

## Что даёт хороший тест

- **Уверенность при рефакторинге** — изменил код, тесты прошли = не сломал
- **Документация** — тест показывает как должен работать код
- **Раннее обнаружение** — ошибка находится при разработке, не в production

---

## Code Coverage — покрытие кода

**Покрытие** — процент строк кода который выполняется во время тестов.

- 0% — тестов нет
- 80% — хорошая цель для большинства проектов
- 100% — часто избыточно и дорого

CI может упасть если покрытие ниже минимума.
