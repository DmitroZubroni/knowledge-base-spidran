## Монолит — всё в одном

Всё приложение разворачивается как один процесс. Авторизация, каталог, оплата, уведомления — один деплой.

```
┌──────────────────────────────┐
│          Монолит             │
│  ┌────────┐  ┌────────────┐  │
│  │ Users  │  │  Products  │  │
│  ├────────┤  ├────────────┤  │
│  │ Orders │  │Notifications│ │
│  └────────┘  └────────────┘  │
└──────────────┬───────────────┘
               ↓
         [Одна БД]
```

**Плюсы:**
- Просто разрабатывать и отлаживать
- Нет сетевых вызовов между модулями
- Легко деплоить — один артефакт

**Минусы:**
- При росте — тяжело поддерживать
- Одна ошибка может положить всё приложение
- Масштабировать можно только целиком

---

## Микросервисы — каждая функция отдельно

Каждый домен — независимый сервис со своей базой, деплойментом и командой.

```
[User Service] ←→ [Order Service] ←→ [Payment Service]
      ↓                  ↓                   ↓
  [Users DB]        [Orders DB]         [Payments DB]
                         ↕
                  [API Gateway]
                         ↕
                      Клиенты
```

**Плюсы:**
- Масштабировать только нагруженные сервисы
- Независимые деплойменты — обновил Payment не трогая Orders
- Разные команды работают независимо

**Минусы:**
- Сложная инфраструктура
- Отладка сложнее — ошибка может быть в любом сервисе
- Сетевые вызовы медленнее локальных

---

## Правило: начинай с монолита

Netflix, Uber, Amazon — используют микросервисы. Но все они **начинали с монолита**.

Переход на микросервисы оправдан когда монолит реально причиняет боль: долгие деплойменты, конфликты между командами, невозможность масштабировать нужную часть.
