## Классическая проблема

"У меня работает, а на сервере нет."

Разработчик пишет код на Mac с Java 17, а на production-сервере стоит Java 11 и другая ОС. Или на одном сервере нужно запустить два приложения, которые конфликтуют по версиям библиотек.

**Docker** решает эту проблему.

---

## Контейнер — коробка с приложением

**Контейнер** — изолированная упаковка, которая содержит:
- Само приложение
- Все его зависимости (библиотеки, конфиги)
- Нужную версию рантайма (Java, Node.js...)

Запускаешь контейнер — приложение работает одинаково на любом компьютере или сервере, где установлен Docker.

> Аналогия: корабельный контейнер. Внутри может быть что угодно, но стандартный размер позволяет грузить его на любой корабль, поезд или грузовик.

---

## Image и Container

**Docker Image** — шаблон (как класс в программировании). Неизменяемый файл с описанием что должно быть в контейнере.

**Container** — запущенный экземпляр image (как объект). Можно запустить 10 контейнеров из одного image.

```
Dockerfile → docker build → Image → docker run → Container
```

**Dockerfile** — текстовый файл с инструкциями: "возьми Java 17, скопируй мой JAR, запусти вот такой командой".

---

## Docker Compose — несколько контейнеров вместе

Реальное приложение = несколько контейнеров: сервис + база данных + кэш.

```yaml
# docker-compose.yml
services:
  app:
    image: my-app:1.0
    ports: ["8080:8080"]
  postgres:
    image: postgres:15
  redis:
    image: redis:7
```

Одна команда `docker compose up` — все три сервиса запускаются вместе.

---

## Kubernetes — когда одного Docker мало

Когда контейнеров становится много (десятки, сотни), нужен **Kubernetes (K8s)** — система оркестрации. Она следит, чтобы нужное количество контейнеров всегда работало, перезапускает упавшие, распределяет нагрузку.

---

## Итог

- Docker = упаковать приложение со всем окружением
- Image = шаблон, Container = запущенный экземпляр
- Docker Compose = запустить несколько контейнеров вместе
- Kubernetes = управление большим количеством контейнеров
