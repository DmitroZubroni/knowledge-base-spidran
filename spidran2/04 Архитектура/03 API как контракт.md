## API — договор между частями системы

Когда фронтенд и бэкенд пишут разные команды — им нужно договориться: "какие запросы я принимаю и что возвращаю". Это и есть **API как контракт**.

Если бэкенд изменил формат ответа без предупреждения — фронтенд сломается. Контракт нарушен.

---

## Версионирование API

Когда нужно изменить API не сломав существующих клиентов — вводят версию:

```
/api/v1/users    ← старые клиенты используют это
/api/v2/users    ← новые клиенты с новым форматом
```

## Документация API — OpenAPI / Swagger

Стандарт описания REST API — **OpenAPI (Swagger)**. Это YAML/JSON файл который описывает все эндпоинты:

```yaml
paths:
  /users/{id}:
    get:
      summary: Получить пользователя по id
      parameters:
        - name: id
          in: path
          required: true
      responses:
        200:
          description: Пользователь найден
        404:
          description: Не найден
```

Из этого файла автоматически генерируется интерактивная документация (Swagger UI).

## GraphQL — альтернативный подход

**GraphQL** — альтернатива REST где клиент сам указывает что именно хочет получить:

```graphql
query {
  user(id: 42) {
    name
    email
    orders {
      product
      price
    }
  }
}
```

**Плюс:** клиент получает ровно те поля которые нужны — не больше, не меньше.  
**Минус:** сложнее на бэкенде, сложнее кешировать.

Используют Facebook, GitHub, Shopify.
