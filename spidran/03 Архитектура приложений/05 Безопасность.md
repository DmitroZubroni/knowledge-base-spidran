## Аутентификация vs Авторизация

Две разные вещи, которые часто путают:

- **Аутентификация** — "Кто ты?" (логин/пароль, Google OAuth)
- **Авторизация** — "Что тебе можно?" (admin может удалять, user — только читать)

---

## Как работает авторизация с токеном

Современные приложения не хранят сессию на сервере. Используют **JWT (JSON Web Token)**:

```
1. Пользователь логинится → сервер выдаёт JWT токен
2. Токен хранится в браузере (localStorage или cookie)
3. Каждый запрос содержит токен в заголовке:
   Authorization: Bearer eyJhbGc...
4. Сервер проверяет подпись токена → знает кто это
```

JWT — это просто строка в формате `header.payload.signature`. Внутри: id пользователя, роли, срок действия.

---

## Хеширование паролей

Пароли **никогда** не хранятся в открытом виде. Только хеш:

```
"password123" → bcrypt → "$2a$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy"
```

Хеш нельзя обратно расшифровать. При проверке — хешируется введённый пароль и сравнивается с сохранённым.

---

## HTTPS — шифрование в transit

Все данные между клиентом и сервером должны идти через HTTPS (TLS). Без этого:
- Провайдер видит твои запросы
- Хакер в публичной Wi-Fi сети может перехватить токены

Сегодня HTTPS — это норма, не опция. Let's Encrypt даёт бесплатные сертификаты.

---

## CORS — защита браузера

**CORS (Cross-Origin Resource Sharing)** — браузер по умолчанию запрещает сайту `evil.com` делать запросы к `bank.com`.

Если бэкенд хочет разрешить запросы с конкретного фронтенда — он добавляет заголовок:
```
Access-Control-Allow-Origin: https://myapp.com
```

Если видишь ошибку CORS в DevTools — значит бэкенд не настроил это разрешение.

---

## 🎯 Главное

> Аутентификация = кто ты, авторизация = что тебе можно. JWT — стандарт токенов. Пароли только в виде хешей. HTTPS везде. CORS настраивается на бэкенде.
