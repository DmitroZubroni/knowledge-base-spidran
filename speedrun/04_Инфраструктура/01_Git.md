## Проблема без Git

Представь: три разработчика работают над одним файлом. Один сохранил изменения — затёр правки другого. Кто что менял — непонятно. Вернуться к рабочей версии недельной давности — невозможно.

**Git** решает все эти проблемы.

---

## Что такое Git

**Git** — система контроля версий. Она запоминает каждое изменение в коде: кто, что, когда и зачем изменил.

**GitHub / GitLab / Bitbucket** — онлайн-платформы для хранения Git-репозиториев. GitHub — самая популярная.

---

## Ключевые концепции

**Репозиторий (repo)** — папка проекта под управлением Git. Хранит всю историю изменений.

**Коммит (commit)** — сохранённый снимок изменений с сообщением: "добавил авторизацию", "исправил баг #42".

**Ветка (branch)** — параллельная версия кода. Ты работаешь в своей ветке, не мешая другим.

**Merge / Pull Request** — слияние изменений из одной ветки в другую. Обычно проходит через код-ревью.

---

## Как выглядит типичный процесс

```
main (стабильная версия)
  ↓
feature/login (ты работаешь здесь)
  ↓ коммиты
Pull Request → код-ревью → merge в main
  ↓
деплой на production
```

---

## Почему это важно

- **История** — всегда можно вернуться к любой версии
- **Параллельная работа** — 50 разработчиков не мешают друг другу
- **Ревью** — код проверяется перед попаданием в продакшн
- **Ответственность** — видно кто что менял

---

## Итог

Git = инструмент для совместной работы с кодом и хранения истории изменений. Без него не работает ни одна серьёзная команда разработки.
