## Почему не просто файлы

Файл в файловой системе — это линейная запись. Найти конкретного пользователя из миллиона — перебрать всех. **База данных** оптимизирована для хранения и поиска: находит нужную запись за миллисекунды благодаря **индексам**.

---

## Реляционные базы данных

**SQL-базы** (PostgreSQL, MySQL, SQLite) — данные в таблицах со связями между ними.

```
Таблица users:           Таблица orders:
┌────┬──────────┐        ┌────┬─────────┬──────────┐
│ id │ name     │        │ id │ user_id │ product  │
├────┼──────────┤        ├────┼─────────┼──────────┤
│  1 │ Дмитрий  │        │  1 │    1    │ Курс     │
│  2 │ Мария    │        │  2 │    1    │ Книга    │
└────┴──────────┘        └────┴─────────┴──────────┘
```

`user_id` в orders — **внешний ключ**, ссылается на `id` в users.

**SQL** — язык запросов:
```sql
SELECT u.name, o.product
FROM users u
JOIN orders o ON u.id = o.user_id
WHERE u.id = 1;
```

## NoSQL — когда нет фиксированной схемы

**MongoDB** — хранит документы (как JSON). Нет жёсткой схемы таблицы:

```json
{
  "_id": "abc123",
  "name": "Дмитрий",
  "address": { "city": "Москва" },
  "tags": ["developer", "java"]
}
```

**Redis** — хранит данные в памяти (RAM). Молниеносно быстро. Используется для кеша, сессий, счётчиков.

---

## Индексы — почему поиск быстрый

**Индекс** — отдельная структура данных (обычно B-дерево) которая позволяет быстро найти записи по определённому полю.

```
Без индекса: проверить все 10 миллионов записей → секунды
С индексом по email: перейти сразу к нужному → миллисекунды
```

Цена: индекс занимает место и замедляет запись. Создают индексы на поля по которым часто ищут.

---

## Когда что использовать

| Ситуация | Выбор |
|----------|-------|
| Структурированные данные, связи | PostgreSQL |
| Гибкая схема, документы | MongoDB |
| Кеш, сессии, очереди | Redis |
| Аналитика, большие объёмы | ClickHouse |
