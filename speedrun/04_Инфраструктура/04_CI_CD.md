## Проблема ручного деплоя

Раньше выпуск новой версии выглядел так:
- Разработчик собирает проект вручную
- Копирует файлы на сервер по FTP
- Перезапускает сервер вручную
- Молится, чтобы всё заработало

Это медленно, ненадёжно и человеческий фактор постоянно даёт сбои.

---

## CI — Continuous Integration (непрерывная интеграция)

**CI** — автоматическая проверка кода при каждом коммите:

```
Разработчик делает git push
        ↓
CI-система автоматически:
  ✓ Скачивает код
  ✓ Компилирует
  ✓ Запускает тесты
  ✓ Проверяет качество кода (линтер)
  → Результат: OK или ❌ с описанием ошибки
```

Если тесты упали — разработчик сразу узнаёт, до того как код попал в продакшн. Это и называется "непрерывная интеграция" — код постоянно проверяется.

---

## CD — Continuous Delivery/Deployment

**Continuous Delivery** — автоматическая подготовка к деплою. После успешного CI код готов к выкатке, но деплой нажимается вручную.

**Continuous Deployment** — деплой тоже автоматический. Успешный коммит → автоматически попадает на production.

```
Коммит → CI (тесты) → CD (деплой) → Production
```

## Популярные инструменты

| Инструмент | Что делает |
|-----------|------------|
| **GitHub Actions** | CI/CD прямо в GitHub |
| **Jenkins** | Классический CI-сервер, гибкий |
| **GitLab CI** | Встроен в GitLab |
| **ArgoCD** | CD для Kubernetes |

---

## Пример пайплайна

```
1. Разработчик пушит код в ветку feature/login
2. GitHub Actions запускает тесты (2 минуты)
3. Тесты прошли → создаётся Pull Request
4. Код-ревью, approve
5. Merge в main → автоматически деплоится на staging
6. QA тестирует на staging
7. Нажатие кнопки → деплой на production
```

---

## Итог

- CI = автоматическая проверка кода при каждом коммите
- CD = автоматизация доставки кода на сервер
- GitHub Actions — самый простой способ начать
- Цель: выкатывать изменения часто, быстро и без страха
