## Типичное веб-приложение

Возьмём интернет-магазин. Что в нём есть?

```
Пользователь
    ↓ браузер / мобильное приложение
[Frontend]  — то, что видит пользователь
    ↓ HTTP запросы (API)
[Backend]   — бизнес-логика
    ↓
[База данных] — хранение данных
```

Три слоя. Почти любое приложение можно свести к этой схеме.

---

## Слои внутри Backend

Backend обычно тоже делится на слои — это называется **многоуровневая архитектура**:

```
[Controller] — принимает HTTP-запросы, отдаёт ответы
      ↓
[Service]    — бизнес-логика (правила: скидка, проверки, расчёты)
      ↓
[Repository] — работа с базой данных (сохранить, найти, удалить)
      ↓
[Database]   — PostgreSQL, MySQL, MongoDB...
```

Зачем делить? Каждый слой отвечает за своё. Если нужно поменять базу данных — меняешь только Repository, остальное не трогаешь.

---

## Что ещё бывает в приложении

- **Cache (кэш)** — Redis, Memcached. Быстрое хранение часто запрашиваемых данных в памяти, чтобы не ходить в базу каждый раз.
- **Message Queue (очередь)** — Kafka, RabbitMQ. Асинхронное общение между сервисами.
- **CDN** — сеть серверов по всему миру для раздачи статических файлов (картинки, CSS) ближе к пользователю.
- **Load Balancer** — распределяет запросы между несколькими серверами.

---

## Итог

- Приложение = Frontend + Backend + База данных
- Backend делится на слои: Controller → Service → Repository
- Дополнительно: кэш, очереди, CDN, балансировщик
